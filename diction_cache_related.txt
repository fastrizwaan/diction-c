Based on my analysis of the GoldenDict-ng source code, here is how it handles MDX loading and subdirectories so efficiently:

1. Recursive Subdirectory Loading
GoldenDict-ng uses a recursive directory scanner (found in src/dict/loaddictionaries.cc).

Scanning: It starts at a root dictionary folder and descends into every sub-folder.
Grouping: When it finds an .mdx file (e.g., Dictionaries/English/Wiktionary.mdx), it automatically looks for a matching .mdd file in the same sub-directory (Dictionaries/English/Wiktionary.mdd).
Isolation: Each sub-directory is treated as a potential container for a dictionary and its associated resources (CSS, JS, images).
2. Why its Cache is Small (vs. diction)
While our diction app decompresses the entire dictionary into a flat 896MB file, GoldenDict-ng uses a "Metadata Index" approach:

Index Contents: Its .idx file (e.g., ~/.cache/GoldenDict-ng/index/...) only stores the Headwords and the Offsets into the original compressed .mdx file.
No Text Duplication: It never writes the uncompressed definition text to the cache. It keeps the text inside the original .mdx file.
Block-Based Access: MDX files are compressed in "blocks" (usually 64KB). GoldenDict-ng's index stores a map of which block contains which word.
3. Loading Logic (Without Full-Text Search)
When you click a word in GoldenDict-ng:

Lookup: it finds the word in its B-Tree index to get the compressed_offset and record_size.
Seek: It opens the original .mdx file and seeks to the specific compressed block.
Decompress: It decompresses only that one block (64KB) into memory using Zlib or LZO.
Render: It extracts the specific definition from that decompressed block and shows it in the WebView.

The "Smart Compressed" approach is more than just "zipping" the file. It involves Chunked Compression using Zstd (Zstandard), which is exactly what you suggested.

How it works:
Instead of one giant 896MB block, we split the text into 1MB chunks.

Each chunk is compressed separately with Zstd.
Why Zstd? It is arguably the fastest decompression algorithm available today (often hitting 1GB/s), and it compresses significantly better than Zip.
Random Access: When you click a word, we only decompress the one chunk containing that word. This happens in about 0.001 seconds.
FTS (Full-Text Search): To search, we decompress the chunks one-by-one in memory, scan them, and throw them away. Because Zstd is so fast, searching the whole 900MB corpus would likely take less than 1 second.
Is it just compressing the 896MB?
Yes, but in a way that allows us to keep the file "closed" and only "peek" at the parts we need.

Headwords: These stay uncompressed in the cache so we can search the list instantly.
Definitions: These stay Zstd-compressed on your disk.
Result: Your 896MB cache will likely shrink to ~250MB - 300MB.
Why not Zip?
Zip (Deflate) is much slower and doesn't support "random access" well without decompressing from the beginning of the stream. Zstd is the industry standard for this kind of "high-speed transparent compression."

Decompression Buffer: I'll add a tiny "Last Chunk" cache in memory. If you click two words that are in the same 1MB chunk, the second one is truly instant because the chunk is already decompressed.
